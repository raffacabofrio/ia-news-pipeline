## Deferred from: code review of 1-2-worker-pipeline (2026-07-07)

- **Webhook `409 duplicate_in_progress` from the real WordPress receiver is not part of the frozen §5.1 contract and is currently treated as a permanent failure.** The S2.1 receiver (`wp-plugin/ia-pipeline-receiver/ia-pipeline-receiver.php`, `ia_pipeline_create_post_with_idempotency_lock`) can return `409` with `duplicate_in_progress` when two deliveries for the same `job_id` race past its advisory lock (e.g. a message redelivered to a second worker while the first is still finishing its POST). Architecture §5.1 only documents `201`, `200 duplicate:true`, `401`, and `422` as expected responses, and `WordPressWebhookPublisher.PublishAsync` (`service/IaNewsPipeline.Worker/Services/WordPressWebhookPublisher.cs`) falls through to `PermanentFailure` for any unrecognized non-2xx/non-5xx status, so a `409` would incorrectly mark the job `failed` instead of retrying. Fixing this cleanly requires either extending the frozen contract (a `correct-course` decision) or explicitly special-casing `409` as transient in the worker; both are out of scope for S1.2, which is intentionally stub-only per its "Scope boundaries" section. Real end-to-end behavior against the actual receiver belongs to S4.3.
- **No mechanism marks a job `failed` after it exhausts SQS redelivery and lands in the DLQ.** Architecture §3 states "after 5 receives → DLQ + job row `failed` with reason," but this story only implements per-message transient/permanent classification (`JobMessageProcessor.HandleFailureAsync` in `service/IaNewsPipeline.Worker/Services/JobMessageProcessor.cs`) — nothing consumes the `pipeline-jobs-dlq` queue or updates the job row once ElasticMQ/SQS's own redrive policy (`maxReceiveCount: 5`) moves a message off the main queue. A job stuck in transient failure forever will silently stay in `processing`/`publishing` state via `GET /api/jobs/{id}`, with no failure reason ever recorded. This needs a DLQ-consuming component or scheduled sweep, which is new scope beyond S1.2's Tasks/Subtasks and is not covered by its testing requirements either. Candidate home: a later Epic 1 hardening story or the production-path observability work already flagged in architecture §7 ("alarm on DLQ depth").
